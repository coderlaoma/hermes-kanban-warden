# kanban-warden v0.2 Loop Trace and Verification Design

## Purpose

`kanban-warden` v0.2 turns the MVP supervisor into a traceable Loop Supervisor. The MVP already watches Kanban events, stores cursors, applies conservative state-machine actions, and protects shared boards with leader locks and idempotency. v0.2 adds an explicit supervision layer that explains what the warden saw, why it classified a task loop a certain way, what it plans to do, and how it verifies that an action actually changed the task state.

The objective is trust. Operators should be able to inspect a card and understand the warden's reasoning without reading logs or reverse-engineering SQLite rows.

## Scope

v0.2 covers:

- loop state classification per task
- loop trace records for every non-trivial decision
- post-action verification contracts
- no-progress detection for loops that are active but not converging
- dry-run reporting that groups observations and planned actions by card

v0.2 does not cover:

- self-modifying code
- automatic policy tuning
- transport-level notification delivery guarantees
- broad dashboard work beyond structured dry-run/status output

## Loop State Model

The warden keeps its own supervision state instead of overloading Kanban card status.

States:

- `observing`: the card is visible, and no issue is detected.
- `waiting_for_worker`: the card is assigned/running and expected to produce progress.
- `waiting_for_review`: the source card is blocked on reviewer output.
- `action_planned`: the warden has selected an action but is in dry-run or pending apply.
- `action_applied`: the warden applied an idempotent action and is waiting to verify it.
- `verifying_action`: the next sweep is checking whether the action took effect.
- `no_progress`: the card shows activity or elapsed time but no meaningful state movement.
- `needs_human`: retries, verification, or policy confidence failed and manual attention is required.
- `closed`: the loop has reached done/cancelled/terminal state.

## Loop Trace

Each decision writes a trace record. A trace record must be safe to display to an operator and must not include raw secrets.

Fields:

- `trace_id`
- `board_id`
- `task_id`
- `source_task_id`
- `profile_name`
- `loop_state`
- `observed_facts_json`
- `matched_policy`
- `decision`
- `confidence`
- `planned_action_json`
- `verification_contract_json`
- `created_at`

Example:

```json
{
  "task_id": "impl-123",
  "loop_state": "no_progress",
  "observed_facts_json": {
    "last_event_age_seconds": 5400,
    "worker_events_since_last_progress": 18,
    "artifact_count_delta": 0,
    "status_changed": false
  },
  "matched_policy": "worker_execution.no_progress",
  "decision": "notify_origin",
  "confidence": "medium",
  "planned_action_json": {
    "type": "queue_notification",
    "channel": "origin"
  },
  "verification_contract_json": {
    "next_check_seconds": 900,
    "success": "new_progress_event_or_status_change"
  }
}
```

## Post-Action Verification

Every applied action gets a verification contract. The contract defines what must be true on a later sweep.

Verification examples:

| Action | Success condition | Failure behavior |
|---|---|---|
| `create_reviewer_card` | reviewer card exists, link points to source card, assignee matches policy | mark `needs_human` after idempotent retry fails |
| `comment_source_card` | comment exists once and content hash matches planned message | queue retry, then escalate |
| `unblock_source_card` | source card status no longer blocked by review | record verification failure and notify origin |
| `retry_worker` | new heartbeat, event, state transition, artifact, or useful comment appears | mark `no_progress` or `needs_human` based on retry count |
| `queue_notification` | outbox row is queued or delivered by a later drainer | keep retryable outbox state |

Verification results should be written as loop outcomes so later versions can learn from them.

## No-Progress Detection

Stale means time passed. No-progress means the loop is not converging.

Signals:

- no new task event after the stale threshold
- many worker events without status change or artifact
- repeated identical failure summaries
- repeated reviewer `needs_changes` without source update
- retry actions that return to the same state
- notification retry exhaustion without acknowledgement

v0.2 should implement no-progress conservatively. When confidence is low, report the trace in dry-run but avoid mutation.

## Dry-Run Report

Dry-run output should be grouped by card:

```json
{
  "cards": [
    {
      "task_id": "impl-123",
      "loop_state": "no_progress",
      "observations": ["18 worker events", "no artifact delta", "status unchanged"],
      "planned_actions": ["notify_origin"],
      "verification": "expect progress event within 15m",
      "confidence": "medium"
    }
  ]
}
```

This is the main operator interface for v0.2. It should be readable enough that a user can decide whether to enable apply mode.

## Data Storage

Store traces and outcomes in the warden state database. Reuse the existing state DB path and idempotency approach.

Suggested tables:

- `loop_trace`
- `loop_outcome`

Both tables should have stable IDs and timestamps. JSON fields should be deterministic enough for tests and diffable dry-run output.

## Tests

Minimum tests:

- classify review-required card as `waiting_for_review`
- classify running card without recent events as `no_progress`
- classify high-activity card without artifacts as `no_progress`
- write loop trace once per decision
- verify reviewer-card creation success
- verify failed action escalates to `needs_human`
- dry-run output groups observations by card
- idempotent rerun does not duplicate trace-linked actions

## Rollout

1. Add trace/outcome schema behind migration-safe initialization.
2. Add classification without applying new mutations.
3. Add dry-run grouped report.
4. Add verification contracts for existing actions.
5. Enable no-progress detector in report-only mode.
6. Allow apply mode only for existing low-risk actions.
