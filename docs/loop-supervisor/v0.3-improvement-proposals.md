# kanban-warden v0.3 Improvement Proposal and Config Change Design

## Purpose

`kanban-warden` v0.3 makes the Loop Supervisor improve its policies from evidence. v0.2 records loop traces and outcomes. v0.3 aggregates those records into improvement signals, creates policy/config proposals, and routes low-risk changes through human approval and audit.

The goal is controlled adaptation. The warden can become more accurate as it is used, but it should prefer config and policy changes over code changes.

## Scope

v0.3 covers:

- improvement signal aggregation
- policy/config proposal generation
- human approval records
- audited config patch application
- before/after dry-run comparison

v0.3 does not cover:

- code patch generation
- branch/MR creation
- automatic production rollout without approval
- hidden policy changes

## Improvement Signals

An improvement signal is a durable, evidence-backed observation that current behavior can be better.

Signal types:

- `false_positive`: a policy repeatedly alerts but users ignore or override it.
- `missed_detection`: users manually flag a task that warden did not catch.
- `verification_failure`: an action repeatedly fails post-action verification.
- `retry_ineffective`: retries do not change loop outcomes.
- `threshold_mismatch`: timing thresholds do not match observed team behavior.
- `policy_gap`: repeated pattern cannot be expressed by current config.

Fields:

- `signal_id`
- `signal_type`
- `scope`
- `severity`
- `supporting_trace_ids_json`
- `supporting_outcome_ids_json`
- `summary`
- `recommended_level`
- `created_at`

## Proposal Types

v0.3 should focus on E1 and E2 proposals.

### E1 Recommend

The warden creates a recommendation but does not patch files.

Example:

```yaml
type: recommend
target: policies.review_loop.reviewer_timeout
current: 2h
suggested: 6h
reason: 83% of alerts were manually ignored over the last 7 days
```

### E2 Config Change

The warden prepares a config patch and applies it only after approval.

Example:

```yaml
type: config_change
target_file: config.yaml
patch:
  kanban_warden.policies.review_loop.reviewer_timeout: 6h
approval_required: true
```

## Human Approval

Every config-changing proposal must include:

- proposal ID
- evidence summary
- exact config path and current value
- proposed value
- expected behavior change
- risk level
- rollback value
- approval actor
- approval timestamp

Approval can be recorded through a CLI command, a Kanban card action, or a future chat workflow. The storage model should not assume one specific UI.

## Audit Trail

Each proposal writes audit events:

- `proposal_created`
- `approval_requested`
- `human_approved`
- `config_patch_prepared`
- `dry_run_before`
- `config_patch_applied`
- `dry_run_after`
- `rollback_prepared`
- `proposal_closed`

Audit entries must contain the proposal ID, actor, timestamp, and sanitized payload.

## Config Patch Rules

Rules:

- Patch only whitelisted `kanban_warden` policy keys.
- Preserve comments when practical; otherwise include a clear audit diff.
- Back up the target config before writing.
- Never patch secrets or provider credentials.
- Never patch plugin install paths without v0.4 approval.
- Never restart a live gateway unless the approval scope includes restart.

## Before/After Evaluation

For every config proposal, v0.3 should run dry-run before and after the patch against the same fixture or recent event window.

Report:

```json
{
  "proposal_id": "imp-001",
  "before": {
    "alerts": 18,
    "false_positive_estimate": 0.83
  },
  "after": {
    "alerts": 4,
    "false_positive_estimate": 0.25
  },
  "changed_policies": ["review_loop.reviewer_timeout"]
}
```

If before/after cannot be computed, the proposal must say so and require stricter approval.

## Example

```yaml
proposal_id: imp-2026-06-review-timeout
level: E2
signal_type: false_positive
title: Raise reviewer timeout from 2h to 6h
evidence:
  traces: 18
  manually_ignored: 15
  false_positive_rate: 0.83
current:
  policies.review_loop.reviewer_timeout: 2h
suggested:
  policies.review_loop.reviewer_timeout: 6h
risk: low
rollback:
  policies.review_loop.reviewer_timeout: 2h
```

## Tests

Minimum tests:

- aggregate false-positive signal from loop outcomes
- aggregate verification-failure signal from failed action checks
- create recommendation proposal without modifying config
- create config-change proposal with exact patch
- reject proposal for non-whitelisted config path
- record approval and audit events
- apply approved config patch with backup
- run before/after dry-run comparison
- rollback approved config patch

## Rollout

1. Add signal and proposal tables.
2. Generate E1 recommendations only.
3. Add approval storage.
4. Add whitelisted E2 config patching.
5. Add dry-run before/after comparison.
6. Enable config apply for low-risk policies only.
